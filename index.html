Contexto
O que está mudando no WhatsApp
Uma mudança histórica na forma como empresas identificam seus clientes na plataforma.
O cenário atual

Hoje, quando um cliente fala com sua empresa pelo WhatsApp, o sistema identifica ele pelo número de telefone. Esse número é a "chave" que conecta o usuário ao histórico de atendimento, ao CRM e a todos os fluxos automatizados.

A Meta (empresa dona do WhatsApp) está mudando isso. Usuários podem criar um @username e escolher não compartilhar o número de telefone com empresas. Para esses usuários, o sistema precisa de um novo identificador.

O que pode parar de funcionar
  • ✕Fluxos que capturam o telefone automaticamente — podem receber o campo vazio ou com um identificador que não representa o número de telefone
  • ✕Busca de clientes no CRM por número — pode não encontrar ninguém
  • ✕Disparos que dependem do telefone — podem falhar silenciosamente
  • ✕Duplicação de contatos — o mesmo cliente aparecendo duas vezes
O que garante a continuidade
  • ✓Adotar o BSUID como identificador principal — presente em 100% das interações
  • ✓Mapear a base existente enquanto telefone e BSUID ainda chegam juntos
  • ✓Adaptar fluxos e integrações com lógica de fallback antes de agosto
  • ✓Salvar o BSUID no perfil de cada contato para consultas futuras
⚠
Por que agir agora?
Existe uma janela de coexistência — um período em que o telefone e o BSUID chegam juntos nas interações. Quando essa janela fechar, usuários sem telefone chegam sem nenhuma referência anterior. Quanto mais cedo o mapeamento for feito, maior a cobertura da base.

💡
Como ler o BSUID na prática
O BSUID tem formato como BR.11815799212886844830. O prefixo indica o país. Cada empresa recebe um BSUID diferente para o mesmo usuário — o BSUID do seu cliente na sua empresa não é o mesmo que ele teria em outra empresa.

O problema que este pacote resolve

Hoje existe um script padrão que converte o campo de identidade do contato em número de telefone e o salva no perfil. Com a mudança da Meta, esse campo pode chegar vazio ou conter um identificador que não é telefone — como um GUID de 32 caracteres. Sem atualização, o script vai salvar esse valor inválido no lugar do número, corrompendo os dados de todos os contatos que não compartilharem o telefone.

Este pacote substitui esse script pelo novo campo nativo da Meta (waId), que traz o telefone quando disponível e deixa o campo vazio quando não. A partir daí, o dado salvo é sempre válido — telefone real ou campo em branco.

O que é feito
  • ✓Atualização do script de identificação de contatos para compatibilidade com o novo modelo da Meta
  • ✓Garantia de que o campo de telefone passa a ser preenchido corretamente — com o número quando disponível, ou vazio quando não compartilhado
  • ✓Os dados existentes no perfil do contato são preservados — nenhuma informação anterior é perdida
  • ✓Gestores passam a poder realizar disparos de mensagens por BSUID diretamente no Blip Desk
O que NÃO está incluído
  • ✕Migração retroativa de contatos já existentes sem BSUID
    (coberto pelo Pacote 1)
  • ✕Adaptação de fluxos que usam o telefone para busca externa
    (coberto pelo Pacote 2)
  • ✕Alteração de qualquer lógica de negócio além do salvamento da variável

O problema que este pacote resolve

Sua base tem milhares de registros com telefone, mas sem BSUID. O Pacote 0 começa a salvar o BSUID a partir de agora — mas e quem já está na base e não vai interagir com o bot nos próximos meses?

A Blip disponibilizou um endpoint (canal de consulta técnica) que permite enviar uma lista de telefones e receber os BSUIDs — sem disparar mensagem e sem custo por usuário. Este pacote executa essa consulta para toda a base.

O que é feito
  • ✓Consulta da base existente do cliente para identificar o BSUID de cada contato já cadastrado
  • ✓Enriquecimento dos registros no CRM com o novo identificador, sem necessidade de disparar mensagens ou interromper operações
  • ✓Entrega de relatório com o resultado do mapeamento: quem foi identificado e quem permanece sem BSUID — servindo de base para o Pacote 3, se necessário
⚠
Limitações importantes — leia antes de contratar
O endpoint não consulta a Meta em tempo real. Ele retorna apenas os relacionamentos que a Blip já recebeu e armazenou. Se um usuário nunca interagiu com o bot desde abril de 2026, o endpoint não vai retornar nada para esse número — sem erro, sem aviso.

Cobertura esperada: 70–80% para bases muito ativas. Para bases com muitos inativos, pode ser bem menor. Os números sem retorno são cobertos pelo Pacote 3, se necessário.

Requisitos obrigatórios do cliente: o campo de BSUID já deve ter sido criado no CRM antes da execução. O sistema do cliente precisa ter API REST disponível — tanto para consulta por telefone (GET) quanto para atualização do novo campo (POST). CRMs sem API não são compatíveis com este pacote.

⚠
Quando usar este pacote
O Pacote 3 só faz sentido após os Pacotes 1 e 2 concluídos. Ele cobre os usuários que o endpoint não retornou — aqueles que não interagiram recentemente e que, por razão crítica de negócio, precisam estar mapeados antes de agosto. Para a maioria dos clientes, Pacotes 0+1+2 já são suficientes.
O problema que este pacote resolve

Após o mapeamento em batch, parte da base permanece sem BSUID — usuários inativos que nunca interagiram desde o início do rollout. A única forma de capturar o BSUID desses usuários é provocar uma interação.

Este pacote faz isso de forma inteligente: um disparo com conteúdo relevante para o usuário — não uma mensagem técnica — que gera a interação e captura o BSUID automaticamente.

O que é feito
  • ✓Identificação dos contatos que permanecem sem BSUID após o Pacote 1 e definição do segmento prioritário para o disparo
  • ✓Alinhamento com o cliente sobre o conteúdo da mensagem — o disparo precisa ter valor real para o destinatário
  • ✓Execução do disparo e captura automática do BSUID dos usuários que responderem
  • ✓Entrega de relatório com o resultado: cobertura adicional obtida e contatos que permaneceram sem retorno
O que NÃO está incluído
  • ✕Custo das mensagens disparadas — cobrado por mensagem pela Blip/Meta e repassado diretamente ao cliente
  • ✕Criação de conteúdo de marketing — definido pelo cliente
  • ✕Cobertura de usuários que bloquearam a empresa

Referência
Glossário de termos
Todas as siglas e termos técnicos usados neste guia, explicados em linguagem simples.
BSUID
Business-Scoped User ID
O novo identificador principal de usuários no WhatsApp, criado pela Meta. Cada usuário tem um BSUID diferente para cada empresa com que interage. Ao contrário do telefone, ele é garantido em 100% das interações e não depende de configurações de privacidade.
GUID
Globally Unique Identifier — ID Blip interno
Código interno gerado pela Blip para identificar contatos quando o telefone não é compartilhado. Tem 32 caracteres e não pode ser usado para buscar o cliente em sistemas externos. Formato: e4b11bdd-a9bf-46ad...@wa.gw.msging.net
PN
Phone Number — Número de telefone
O identificador tradicional do WhatsApp. Continua existindo, mas deixa de ser garantido para todos os usuários. Quem ativa um @username pode optar por não compartilhá-lo com empresas.
CRM
Customer Relationship Management
Sistema de gestão do relacionamento com o cliente. Exemplos: RD Station, HubSpot, Salesforce. É onde ficam os dados dos contatos. Com a transição, precisa ganhar um campo novo para o BSUID.
API
Application Programming Interface
Canal de comunicação entre sistemas. Quando o Blip "conversa" com o CRM do cliente, faz isso por uma API. O endpoint PN→BSUID é um recurso da API da Blip que permite consultar BSUIDs em lote.
Endpoint
Ponto de acesso de uma API
Um endereço específico dentro de uma API que aceita um tipo de consulta. O endpoint PN→BSUID da Blip recebe uma lista de telefones e retorna os BSUIDs conhecidos. É acessado via código, não manualmente.
Metadata
Metadados — informações extras da mensagem
Cada mensagem que chega no Blip vem com um pacote de informações invisíveis ao usuário. É nesse pacote que o BSUID viaja. O campo específico é #wa.bsuid.
Fallback
Plano alternativo automático
Quando o sistema tenta usar uma informação e não encontra, o fallback é o comportamento alternativo. Exemplo: "se não existe telefone, usa o BSUID; se não existe BSUID, pede ao usuário". Torna o sistema resiliente à ausência de dados.
Rollout
Implantação gradual de uma mudança
A Meta não ativa os usernames para todos de uma vez — faz em fases, por região e perfil de usuário. No Brasil, o rollout de usernames começa em agosto de 2026.
Builder / Studio
Ferramentas de criação de fluxos do Blip
Ambientes dentro da plataforma Blip onde os fluxos de conversa são criados e editados. É onde ficam os blocos, as condições, os scripts e as ações — incluindo a ação "Definir contato" usada para salvar variáveis no perfil do usuário.
Rate limit
Limite de velocidade de requisições
A API da Blip tem um limite de quantas consultas podem ser feitas por segundo. Quando esse limite é atingido, as requisições precisam ser repetidas com um intervalo de espera. Por isso o mapeamento em lote exige controle de cadência nas chamadas.
BM
Business Manager — Portfólio Empresarial da Meta
Conta empresarial no ecossistema Meta que agrupa páginas, números de WhatsApp e contas de anúncio. O BSUID é definido no nível do BM — por isso o mesmo usuário tem BSUIDs diferentes para empresas em BMs diferentes.
